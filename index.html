    API Reference

    Introduction

    Documentation for the GDS API.

    GDS is implemented as a REST-style API that accepts and returns JSON. It can be accessed at:

    https://api.storage

    API Examples

    The main body of documentation is intended to describe the URLs, parameters, and response for interacting with GDS.

    Reference examples of requests and replies from the API are provided in the right hand column. They are meant to describe the call but will not necessarily be exhaustive.

    Variables

    Variables are represented by <variable_name> in both the documentation and examples. Whenever a variable is required it will be defined in the body of the documentation.

    Parameter Description   
    id the facility ID required
    liveData query the PMS in real time

    if a variable is not listed as required it is optional and may be omitted.

    Data Model

    API requests that require a body will have the elements defined in the Request Body table.

    Element Description   
    address root element required
    ❯ address1 fist line required
    ❯ address2 second line
    ❯ city city required
    ❯ stateCode state required
    ❯ postalCode postal code required

    if not listed as required input parameters are optional and may be omitted.

    PMS “raw” Data

    Most return objects from GDS will include a pmsRaw map. This is the raw data returned from the pms before being parsed into the GDS data model.

    Caching

    GDS implements a cache between the PMS and API users. By default, cached data will be sent whenever possible. This can be overridden in most cases by passing livedata=true as a query parameter which will force GDS to query the PMS.

    Most data objects and some api calls will have a lastUpdate response key that contains a timestamp indicating the last time the data was refreshed.

    GDS will automatically refresh its cache based on the lastUpdate timestamp as it returns from each API call.

    Using the API

    Versioning

    GDS follows major.minor.patch semantic versioning. The version number indicates:

    MAJOR Incremented when breaking api changes are implemented
    MINOR Incremented when new backward compatible changes are implemented
    PATCH Incremented when bug fixes or minor refactoring is pushed live

    The major version must always be provided as a part of the request URL and prefixed with the letter v. For example, version 1 of the api is located at:

    https://api.storage/v1/

    It is not possible to access a specific minor or patch version.

    The current api version will always be returned in the x-storageapi-version header:

    x-storageapi-version: 1.2.3

    Authentication

    Access to GDS is controlled via an API key which is provided in the x-storageapi-key header of every request.

    x-storageapi-key: b594439165d24e23815b9ce79b64aeed

    Required Headers

    There are two headers that are required to be sent in every API request:

    Header Description
    x-storageapi-key your api key
    x-storageapi-date unix timestamp indicating when your generated the request

    x-storageapi-date cannot be in the future or older than 5 minutes. All timestamps are UTC.

    Put a hold on a space. When a space is held, GDS will refuse to rent or reserve it unless the appropriate hold token is provided at the time of the call.

    If the underlying PMS does not support hold functionality this will only guarantee that GDS refuses to rent or reserve a unit. This could result a failed rental or reservation if the space is taken out of inventory by the PMS directly

    A successful call to this endpoint will return a hold token that expires after a specific length of time. Until expired, this token must be provided when renting or reserving the unit otherwise the operation will fail.

    Held spaces are not considered available.

    Error returns from GDS follow the same general pattern as all responses, with a message and data section. The data section will contain the errors key that has the following fields:

    Key Description
    code the main error code indicating what failed
    origin an object generated by the process that failed to complete
    ❯ message a user friendly message indicating what happened
    ❯ code error code from the originating source
    ❯ details any additional details about the error for troubleshooting purposes
    ❯ origin an error object that contains more information about the error

    When an error occurs the HTTP status will provide a broad indication of what went wrong.

    Code Description
    400 request invalid
    404 resource not found
    500 server error

    Error Codes

    Error Codes are meant to be descriptive of what actually took place and in many cases you can deduce the problem from the code alone.

    GDS error codes

    Code Description
    PermissionDenied API key does not have access to the resource or permission to perform the action attempted
    ResourceLocationInvalid The resource you are attempting to access is invalid.
    RequestInvalid there was a problem with the request, for example a missing required header
    APIKeyInvalid key is invalid - either ill formatted or could not be found
    APIKeyDisabled key has been disabled
    ValidationError provided data is invalid. The detail key will contain more information about what piece of data is incorrect and why

    Root call error codes

    Calls will have a root error that matches the call indicating it failed. For example, the call to get facility data, if it errors, will return the GetFacilityFailed error code.

    PMS call error codes

    When the origin is an error coming from the PMS integration, it is usually because there was a problem communicating with the PMS. For example, MakePaymentCallError indicates that the response from the PMS was not what was expected or did not succeed. With these errors, the details key in the map will have more information about what PMS api call actually failed. Further, if your API key is in debugging mode, you will be able to access the actual request/response data from the PMS via the debugging dashboard.
